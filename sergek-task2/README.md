# Задача 2 — Рассыльщик WhatsApp (демонстрация принципа)

## Состав

- `whatsapp_sender.py` — основной скрипт (режимы `dry-run` и `live`)
- `prepare_messages.py` — превращает выгрузку парковок из задачи 1
  (`almaty_parkings.csv`) в таблицу `phone_number,message`
- `messages_sample.csv` — пример входной таблицы (3 «парковочных» строки +
  2 строки с ошибками для демонстрации обработки)
- `sender_dry_run.log`, `report_dry_run.csv` — результат прогона на примере

## 1. Подготовка таблицы из задачи 1

```bash
python prepare_messages.py \
    --input almaty_parkings.csv \
    --output messages.csv \
    --test-number "+7XXXXXXXXXX"
```

Каждая строка получает один и тот же тестовый номер (используется только
для демонстрации dry-run на полной базе — реально на него ничего не уходит).

## 2. Dry-run (по умолчанию, безопасный режим)

```bash
python whatsapp_sender.py --input messages.csv --mode dry-run
```

Ничего не отправляется. На выходе:

- `sender.log` — построчный журнал (валидация, статус, причины пропуска);
- `report.csv` — таблица со статусом каждой строки: `DRY_RUN_OK` /
  `ERROR (invalid_phone | empty_message)`.

Этот режим можно безопасно запускать на **полной базе из задачи 1** —
именно так показана работа программы.

## 3. Live (1–2 сообщения на свой/тестовый номер)

Используется официальный **WhatsApp Cloud API** (Meta for Developers).
Бесплатный тестовый номер и токен выдаются в личном кабинете разработчика;
получатель тоже должен быть верифицирован как тестовый номер.

```bash
python whatsapp_sender.py \
    --input messages.csv \
    --mode live \
    --max-messages 2 \
    --token "$WA_TOKEN" \
    --phone-id "$WA_PHONE_ID"
```

Скрипт жёстко ограничивает live-режим **двумя сообщениями** независимо от
значения `--max-messages`. Для демонстрации сделайте скринкаст/скриншоты:
запуск команды → лог с `sent=...` → входящее сообщение на телефоне.

## Обработка ошибок (встроена и проверена)

| Ситуация                         | Поведение программы                                            |
|-----------------------------------|------------------------------------------------------------------|
| Невалидный номер телефона          | строка помечается `ERROR/invalid_phone`, обработка продолжается  |
| Пустой текст сообщения             | строка помечается `ERROR/empty_message`, обработка продолжается  |
| Обрыв соединения / таймаут         | перехватывается (`requests.exceptions.*`), пишется в лог, скрипт идёт дальше |
| Лимит запросов (HTTP 429)          | распознаётся отдельно, пауза и повторная попытка                  |
| Прочие ошибки API (4xx/5xx)        | текст ошибки из ответа API пишется в лог, без падения             |
| Непредвиденная ошибка              | перехватывается верхним `try/except`, скрипт завершается штатно   |

В прилагаемом примере `sender_dry_run.log` видно:
- 3 строки корректны (`DRY_RUN_OK`);
- строка с номером `123` помечена `invalid_phone`;
- строка с пустым сообщением помечена `empty_message`;
- итог: `would_send=3, errors=2, total=5`.

Отдельно (см. `sender_live_error_demo.log`) продемонстрирована ситуация
обрыва соединения/ошибки API при live-отправке — скрипт корректно
сообщает причину и не падает.

## Важно: ограничение на массовую рассылку

Скрипт **не предназначен** для массовой рассылки по чужим номерам.
- В dry-run сообщения никуда не уходят — только лог/отчёт.
- В live режиме обработка жёстко ограничена 2 сообщениями и предполагает
  отправку на собственный/тестовый номер пользователя.
